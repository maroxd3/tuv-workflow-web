So funktioniert deine Prüfstelle Stell dir das System wie eine echte Werkstatt vor — mit drei Stationen und drei Waechtern.

1Die drei Stationen

Jeder Klick im Prüfstellen-System geht den gleichen Weg: vom Tablet des Prüfers → zum Empfangs-Mitarbeiter → zum Aktenschrank. Wie in einer echten Werkstatt.

📱

Das Tablet

Was der Prüfer in der Hand haelt
Zeigt die Termine für heute, die Fahrzeuge, die Mängel. Hat Buttons wie "Bestanden", "Mangel hinzufuegen", "Speichern".
Im Code heißt das: Frontend (React)
💁

Der Empfang

Nimmt jede Anfrage entgegen, leitet sie weiter
Wenn der Prüfer auf "Bestanden" tippt, ruft das Tablet beim Empfang an. Der Empfang prüft, ob die Anfrage Sinn macht, und gibt sie an den Aktenschrank weiter.
Im Code heißt das: Express-API (Container 1)
🗄

Der Aktenschrank

Speichert ALLES, was die Prüfstelle weiss
Halter, Fahrzeuge, Termine, Mängel — alle Eintraege in sauberen Ordnern (Tabellen). Wer was wann gespeichert hat, wird mitgeschrieben.
Im Code heißt das: MariaDB (Container 2)
→ →

2Die drei Waechter (gegen Schummel-Eintraege)

Die wichtigste Regel in deiner Prüfstelle: "Ein Termin darf NICHT auf 'Bestanden' gesetzt werden, wenn das Auto noch einen erheblichen (EM) oder gefährlichen Mangel (GfM) hat." Damit das auf KEINEM Weg passieren kann, gibt es jetzt drei Waechter — einen an jeder Station.

Waechter 1
🚫

Tablet-Waechter

Schon im Tablet
Wenn das Auto blockierende Mängel (EM oder GfM) hat, ist der "Bestanden"-Button ausgegraut — der Prüfer kann ihn gar nicht erst drücken.
Im Code: UI-Guard in den Views
Waechter 2
✋

Empfangs-Waechter

Im Empfangs-Mitarbeiter
Selbst wenn jemand das Tablet austrickst und direkt eine "Bestanden"-Anfrage schickt: Der Empfang schaut nochmal in den Aktenschrank, sieht den blockierenden Mangel (EM/GfM) und sagt "Nein, geht nicht."
Im Code: API-Guard in /api/termine/:id/status
Waechter 3 - NEU
🔒

Akten-Waechter

Im Aktenschrank selbst
Selbst wenn jemand AM Empfang vorbei direkt zum Schrank rennt und versucht eine "Bestanden"-Akte reinzulegen: Der Schrank selbst lehnt ab und wirft die Akte raus.
Im Code: MariaDB-Trigger trg_termin_wf01_update — heute eingebaut

3Drei Schummel-Versuche — alle scheitern

Stell dir vor, jemand will einen Termin faelschlich auf "Bestanden" setzen, obwohl 5 Hauptmängel drin sind. Was passiert?

🤥Schummel A — über das Tablet
Prüfer oeffnet das Tablet, sieht das Auto mit 5 Hauptmängeln, will trotzdem auf "Bestanden" tippen.
Der Button ist ausgegraut. Prüfer kann nicht mal drücken.
Gestoppt von: Waechter 1 (Tablet)
🥷Schummel B — mit einem Trick-Befehl
Jemand kennt die App und schickt direkt einen Trick-Befehl an den Empfang ("setz das einfach auf Bestanden, kein Zucken").
Der Empfang prüft, sieht 5 Hauptmängel, antwortet "Verboten."
Gestoppt von: Waechter 2 (Empfang)
🦹Schummel C — direkt zum SchrankNEU
Jemand schleicht sich am Empfang vorbei (Hacker, IT-Mensch, jemand mit Adminzugriff) und will direkt in den Aktenschrank greifen.
Der Schrank selbst sagt "STOPP — §29 StVZO." und wirft die Akte zurück.
Gestoppt von: Waechter 3 (Akten)

4Was Docker damit zu tun hat (vorher vs. heute)

Docker = fertig gepackte Umzugskartons. Du musst nicht mehr 100 Sachen einzeln installieren und konfigurieren. Ein Karton hat den Empfang drin, der andere hat den Aktenschrank drin. Du sagst "auspacken", beide stehen einsatzbereit.

Vorher (umstaendlich)

  1. MariaDB auf dem PC installieren (eine halbe Stunde)
  2. Passwort setzen, Nutzer anlegen, Datenbank anlegen
  3. Node.js installieren
  4. npm install (alle Bibliotheken laden)
  5. Server-Datei starten
  6. Wehe was kollidiert mit anderen Sachen auf dem PC

Heute (1 Befehl)

  1. Docker installieren (einmalig)
  2. In den Ordner gehen und Befehl tippen:
docker compose up -d

Beide Kartons werden ausgepackt, beide Container laufen, App ist erreichbar unter localhost:8787. Auf jedem Kunden-PC genau gleich — kein Konfigurations-Chaos.

5Warum das Frau Fuchs gefallen wird

Frau Fuchs ist Datenbank-Frau. Sie schaut zuerst auf den Aktenschrank.

  • "Wir haben einen sauberen Aktenschrank" (3NF-Schema mit FK-Constraints) — das erwartet sie.
  • "Unsere Aktenschrank-Tueren haben Pruef-Mechanik" (CHECK, UNIQUE, FOREIGN KEY) — gut.
  • "Unser Aktenschrank wirft falsche Akten von SELBST raus" (BEFORE-UPDATE-Trigger mit SIGNAL SQLSTATE) — DAS überrascht sie. Das hebt deine Note Richtung 1,7.
  • "Wir schreiben jede Änderung mit" (Binary Logging für Point-in-Time-Recovery) — Betriebskonzept, nicht nur "wir machen Backup".
  • "Selbst wenn jemand Empfang umgeht, halten Akten-Regeln die Daten sauber" (Defense-in-Depth-Argument) — genau die Sprache, die sie hoeren will.
