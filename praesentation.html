01 · Problem

Warum dieses Projekt?

Die meisten kleinen und mittleren Prüfstellen verwalten ihren Tagesablauf bis heute mit Excel und Papier. Daraus entstehen konkrete operative Schmerzen.

Status quo in der Branche

  • Termine werden in Excel-Kalendern geführt — kein gemeinsamer Stand zwischen Empfang, Prüfern und Leitung
  • Doppelvergabe von Slots, weil Empfang nicht sofort sieht, dass ein Prüfer bereits auf einem Termin sitzt
  • Tippfehler bei Kennzeichen, FIN, Halter-Daten — keine strukturelle Prüfung
  • Mängelerfassung als Freitext-Notizen, ohne Bezug zum StVZO-Anlage-VIII-Katalog
  • Berichte werden in Word kopiert, manuell ausgefüllt, sind nicht GoBD-konform

Vision von TÜV Prüfstelle Pro

Ein zentrales System für den ganzen Prozess

Tagesplan, Fahrzeug-Stammdaten, Mängelkatalog, Workflow-Steuerung und Berichts-Generierung in einer Anwendung — als Web-App und als native Desktop-App (Tauri).

Strukturierte Eingabe statt Freitext

Hersteller, Modell und Fahrzeugtyp als abhängige Auswahllisten. Kennzeichen gegen die KBA-Liste validiert. Mängel aus dem § 29 StVZO-Katalog. Tippfehler werden strukturell ausgeschlossen.

Echtzeit-Synchronisation

Statusänderungen eines Prüfers sind beim Empfang sofort sichtbar — keine Doppel-Vergabe mehr.

02 · Stakeholder

Wer arbeitet mit dem System?

Vier Rollen mit unterschiedlichen Anliegen. Jede UI-Sicht ist auf eine Hauptrolle zugeschnitten.

Prüfingenieur

Führt die HU/AU am Fahrzeug durch. Will: Mängel schnell aus dem Katalog erfassen, Status-Wechsel mit einem Klick (Starten/Fertig).

Sicht: Tagesplan, Mängel-Modal

Prüfstellenleiter

Plant die Auslastung, überwacht Qualität. Will: Tagesübersicht, KPIs, Bestehensquote, Leistung pro Prüfer.

Sicht: Statistik, Tagesplan-Wochenansicht

Empfang / Admin

Pflegt Fahrzeug- und Halter-Daten, vergibt Termine telefonisch. Will: Schnelle Suche, Plausibilitäts-Hinweise bei der Eingabe.

Sicht: Fahrzeuge, Tagesplan-Tabelle

Fahrzeughalter

Indirekter Stakeholder. Will: Verständlichen, gesetzeskonformen Bericht erhalten.

Sicht: Berichts-PDF (in Roadmap), heute Text-Export

Wie schnell muss die App reagieren — und warum?

Performance-Budgets sind keine Bauchgefühle. Sie folgen aus dem Lastprofil und der Forschung zur menschlichen Wahrnehmung.

Reaktionszeit-Schwellen (nach Nielsen)

SchwelleWirkungBewertung
< 50 msVom Mensch nicht von Null-Latenz unterscheidbarÜber-Spec
≈ 100 msAktion fühlt sich „direkt" an — keine Verzögerung spürbarSweet Spot
≈ 200 msSystem reagiert, aber Gefühl direkter Manipulation geht verlorenZu lasch

Konsequenz: NF-PF-02 (Tagesplan-Filter) auf < 100 ms p95 festgelegt — das ist die ergonomische Grenze, ab der Eingabe-Reaktion direkt wirkt.

Test-Methodik (für alle NF-PF-Werte)

1. Synthetik-Lastdaten

500 Termine + 200 Fahrzeuge — das 10-fache der typischen Praxis-Last.

2. Vitest bench()

100 Durchläufe pro Filter-Funktion, p95-Auswertung.

3. React Profiler

Render-Zeit nach setFilter() via performance.now().

4. Definiertes Referenzgerät

Intel i5-8250U / 8 GB RAM / Chrome stable — Mid-Tier-Hardware der Zielgruppe.

Andere NF-PF-Werte

  • Navigation zwischen Views: < 150 ms — Framer-Motion-Transition liegt bei 180 ms, niedriger nicht wahrnehmbar
  • Firestore-Write: < 400 ms — SLA + EU-Latenz
  • Statistik-Aufbau: < 500 ms für 1 000 Termine — Recharts-Budget

06 · Datenmodell

Entitäten, Beziehungen & Integritätsbedingungen

Drei Kern-Entitäten in 1:N-Beziehung. Integrität wird auf vier Ebenen durchgesetzt — nicht nur Kardinalitäten.

ER-Diagramm

FAHRZEUG ────┐ 1 : N ┌──> TERMIN ────┐ 1 : N ┌──> MANGEL └────────────┘ └────────────┘ (eingebettet) FAHRZEUG { TERMIN { id PK id PK kennzeichen UK fahrzeugId FK fin datum, uhrzeit hersteller* modell* art (HU, AU, …) baujahr [1885..year+1] pruefer typ FK→FAHRZEUG_TYPEN status FK→STATUS kmStand [0..3 000 000] notiz besitzer telefon email mängel: Mangel[] hu_faellig } } MANGEL { code text kat behoben }

Designentscheidung: Mangel hat keine eigenständige Existenz — gehört immer zu genau einem Termin. In Firestore 1 Read statt N+1. In SQL wäre eigene Tabelle sauberer (3NF).

Integrität auf 4 Ebenen

1. Entitäts-Integrität

UID-Primärschlüssel · Kennzeichen unique (case-insensitive) · FIN unique

2. Referentielle Integrität

termin.fahrzeugId verweist auf existierendes Fahrzeug · Cascade-Delete

3. Workflow-Integrität

BESTANDEN nicht zulässig bei HM/GM (§ 29 StVZO) · 4-fach abgesichert

4. Wertebereichs-Integrität

Regex (Tel, E-Mail) · Range (kmStand 0–3M, Baujahr 1885–heute+1) · KBA-Liste · Hersteller/Modell-Whitelist · ISO-3779 FIN-Prüfziffer (weich)

07 · Klassendiagramm

Typmodell & Modellierungsentscheidungen

JavaScript hat keine Klassen im klassischen Sinn — das Datenmodell ist als Object-Schema mit PropTypes-Validation umgesetzt. UML-Notation für Lesbarkeit.
«entity» Fahrzeug ───────────────────────── + id : string «PK» + kennzeichen : string «UK» + fin : string <optional> + hersteller : string «required» + modell : string «required» + baujahr : int [1885..year+1] + typ : FahrzeugTyp «FK» + kmStand : int [0..3 000 000] + besitzer : string + telefon : string <Regex> + email : string <Regex> + hu_faellig : Date «entity» Termin ───────────────────────── + id : string «PK» + fahrzeugId : string «FK» + datum, uhrzeit + art : PruefArt + status : Status + mängel : Mangel[] «embedded» ───────────────────────── + statusWechsel(neu) : void guard: kein BESTANDEN bei HM/GM «value» Mangel ───────────────────────── + code : string «StVZO Anlage VIII» + text : string + kat : {OM, LM, EM, HM, GM} + behoben : bool

Modellierungs-Entscheidungen

Fahrzeug + Termin als Entitäten

Beide haben eigenständigen Lebenszyklus — ein Fahrzeug existiert auch nach allen Prüfungen weiter. Klassische 1:N-Beziehung.

Mangel als embedded Value-Object

Hat keine Beziehungen zu anderen Entitäten — gehört immer zu genau einem Termin. In Firestore weniger Reads; in einer SQL-Variante als Tabelle mangel(termin_id, …).

STATUS & MANGEL_KATEGORIE als Enums

Endliche, nicht änderbare Mengen — STATUS hat 7 Werte (GEPLANT, IN_PRUEFUNG, BESTANDEN, …), MANGEL_KATEGORIE 5 (OM/LM/EM/HM/GM).

Halter eingebettet, nicht eigene Entität

Im Prototyp pro Fahrzeug ein Halter — kein Bedarf an Aggregation. Im Produktivbetrieb 3NF (eigene Halter-Tabelle, ein Halter kann mehrere Fahrzeuge besitzen). SQL-Entwurf in datenmodell.md § 6.5.

08 · Persistenz-Entscheidung

NoSQL (Firestore) vs. relationale DB

Eine bewusst kontroverse Architektur-Entscheidung. Beide Optionen wurden gegeneinander abgewogen — hier die Argumente und unsere Bewertung.

Pro RDBMS (PostgreSQL)

  • Schema-Enforcement & Foreign Keys als DB-Constraint
  • SQL-Aggregationen für Reporting (GROUP BY, Window-Functions)
  • Volle ACID-Transaktionen ohne Operationen-Limit
  • Native JOINs statt Client-side-Aggregation
  • Bewährte Migration-Tools (Flyway, Liquibase)
  • On-Premise hostbar — wichtig für Datenschutz

Pro Firestore (gewählt)

  • Echtzeit-Sync via onSnapshot ohne WebSocket-Server
  • Kein eigenes Backend nötig — passend für 2-Personen-Projekt
  • Eingebaute Offline-Persistence
  • Eingebettete Mängel passen zum Zugriffsmuster
  • Auto-Sharding, keine manuelle Replikation
  • Firebase-Ökosystem (Auth, Storage, Functions verfügbar)
Bewertung im Projekt-Kontext: Für den Prototyp eine akzeptable, nicht optimale Wahl. Unsere Daten sind hochstrukturiert — ein RDBMS wäre für den Produktivbetrieb mittelfristig sauberer (3NF, SQL-Reporting, on-premise-fähig). Firestore-Vorteile (Echtzeit, serverlos) überwiegen im Semester-Scope, sind aber kein dauerhaftes Argument bei wachsendem Reporting-Bedarf.
Migrationspfad dokumentiert in datenmodell.md § 6: SQL-Schema (PostgreSQL inkl. Trigger für Workflow-Regel WF-01), Backend-Layer-Vorschlag (tRPC oder PostgREST), Sync-Ersatz via Supabase Realtime oder eigenem WebSocket.

09 · Datenschutz & Sicherheit

DSGVO bei einer Cloud-DB

Personenbezogene Daten in Firestore erfordern aktive Maßnahmen. Was im Prototyp greift, wo Lücken sind, und wie sie produktiv geschlossen werden.

Verarbeitete personenbezogene Daten

  • Halter-Name / Firma
  • Telefon & E-Mail
  • Kennzeichen (Halter-Bezug)
  • FIN (eindeutige Fahrzeug-ID)
  • Indirekt: Mängel-Historie

DSGVO-Klassifikation: Allgemeine personenbezogene Daten (Art. 4 Nr. 1) — keine besonderen Kategorien.

Aktuelle Maßnahmen

  • Firestore EU-Region europe-west1
  • Auftragsverarbeitungsvertrag mit Google nach Art. 28 DSGVO
  • TLS 1.3 zwischen Client & Firestore
  • Cascade-Delete: Halter-Daten verschwinden mit dem Fahrzeug
  • Keine externen Tracker / Analytics

Lücken im Prototyp + Roadmap

  • Auth fehlt — produktiv: Firebase Auth + Rollen (Prüfer / Leiter / Admin)
  • Audit-Log — produktiv: Cloud-Functions-Trigger
  • Backup — produktiv: Firestore-Export täglich
  • Löschkonzept — Halter-Daten nach 10 Jahren (GoBD) anonymisieren
Alternative bei strenger Datenschutz-Anforderung: Selbst-gehostetes PostgreSQL im lokalen Netz der Prüfstelle. Datenmodell und Validierungs-Logik sind portabel — nur die Sync-Schicht müsste durch Polling oder eigenen WebSocket-Server ersetzt werden.

10 · Validierung

Drei Verteidigungslinien gegen ungültige Daten

Defense in Depth: UX-Strukturierung > Hard-Validation > Soft-Warning. Tippfehler werden strukturell ausgeschlossen, nicht erst durch Fehlermeldungen abgefangen.

Linie 1 · UX-Strukturierung

Cascading Dropdowns für Hersteller → Modell → Typ.

  • ~25 Marken im Hersteller-Dropdown
  • Modell-Dropdown filtert auf Marke
  • Typ-Dropdown filtert auf Marke
  • „Sonstiger"-Modus mit Freitext

Effekt: Tippfehler („Vw" vs. „VW") und Mismatches („BMW Polo") sind strukturell unmöglich.

Linie 2 · Hard-Validation

  • Kennzeichen: Format + KBA-Liste (~430 Codes) + Saison-Suffix MM-MM
  • FIN: 17 Zeichen, keine I/O/Q
  • kmStand: 0 ≤ n ≤ 3 000 000
  • Telefon: Regex, keine Buchstaben
  • E-Mail: Standard-Regex
  • Baujahr: 1885 ≤ n ≤ Jahr+1
  • Workflow: kein BESTANDEN bei HM/GM (4-fach)

Linie 3 · Soft-Warning

  • FIN-Prüfziffer ISO 3779: Position 9 muss zur Berechnung passen

Bewusst weich, weil Pre-1981 / Nicht-Nordamerika-Fahrzeuge keine Prüfziffer tragen — Sonderfälle (Oldtimer, Importe) sollen markiert, nicht abgewiesen werden.

Beispiel: BLABLUBB12343... erfüllt das Format, scheitert an der Prüfziffer → Hinweis.

Warum drei Linien? UX schützt 95 % der Eingaben strukturell. Hard-Validation greift im Sonstiger-Modus, bei programmatischen Aufrufen oder beim Editieren von Altdaten. Soft-Warnings markieren grenzwertige Fälle, ohne Sonderfälle zu blockieren. Auch wenn eine Linie umgangen wird, greift die nächste.

11 · Tests

Test-Strategie & Auswahl der Testfälle

Tests sind nicht zufällig — jeder Testfall folgt einem etablierten Verfahren. Coverage ist ein Nebenprodukt, nicht das Ziel.

Testpyramide (nach Mike Cohn)

╱ e2e ╲ wenige, langsam, teuer ╱────────╲ (noch nicht) ╱ Component╲ mittel, RTL + jsdom ╱────────────╲ ╱ Unit Tests ╲ viele, schnell, isoliert ╱────────────────╲ 114 grün ✓

Was wir bewusst nicht testen

BereichBegründung
Framer-Motion-AnimationenVisuell, nicht funktional
Firestore-IntegrationBräuchte Emulator-Setup
Recharts-RenderingVisual-Regression-Setup eigen

Testentwurfsverfahren

1. Äquivalenzklassen-Methode

Eingaben werden in Klassen partitioniert; pro Klasse 1 Repräsentant. Beispiel validateKmStand: gültig-typisch / leer / negativ / nicht-int / zu-groß = 5 Klassen.

2. Grenzwertanalyse

An den Rändern: gerade-innen, Rand, gerade-außen. validateBaujahr: 1884 (out), 1885 (in), Jahr+1 (in), Jahr+2 (out).

3. Entscheidungstabelle

Bei Workflow-Regeln. validateStatusWechsel: 7 Status × 5 Mängel-Konstellationen → 8 nicht-redundante Kombinationen.

4. Regression aus Fehlermeldungen

Jeder identifizierte Bug erhält einen Test, bevor er gefixt wird. Verhindert Wieder-Regression.

Beispiel-Story (US-06, MoSCoW: M)

„Als Prüfer möchte ich daran gehindert werden, unplausible Hersteller-Modell-Typ-Kombinationen einzugeben (z. B. ‚BMW Polo'), damit Stammdaten konsistent bleiben."

Prio: Must · SP: 8 · Sprint 5

Definition of Done (10 Punkte, teamübergreifend)

  1. Alle Akzeptanzkriterien der Story erfüllt
  2. Code in Feature-Branch, Pull Request erstellt
  3. Mindestens ein Reviewer hat approved
  4. npm run lint — 0 Errors
  5. npm run test — alle Tests grün
  6. Neue Business-Regel = neue Vitest-Tests (Äquivalenz + Grenzwerte)
  7. Keine hartkodierten Strings — Konstanten-Module
  8. PropTypes-Validation für neue Komponenten
  9. Merge in main per Squash-Merge
  10. Manueller Smoke-Test im Browser & Tauri-Desktop-Build
Velocity-Bilanz: 117 SP über vier produktive Sprints (Sprint 2–5), durchschnittlich 29 SP/Sprint. Sprint 5 (Validierungs- + Doku-Sprint) lag bei 50 SP — Schwerpunkt-Sprint.
